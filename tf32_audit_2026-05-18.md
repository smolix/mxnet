# TF32 Math-Type Audit — cuDNN 9 / sm_120 (2026-05-18)

## Background

cuDNN 9 deprecated the legacy `cudnnSetConvolutionMathType` + `CUDNN_TENSOR_OP_MATH_ALLOW_CONVERSION`
v7-style path in favour of the cuDNN frontend / v8 "engine heuristic" API that is now the
primary path in `src/operator/cudnn_ops.cc`.  On Ampere+ (and especially sm_120 / Blackwell)
the expected defaults are:

| Input dtype | Expected math type (cuDNN 9 best practice) |
|-------------|---------------------------------------------|
| FP32        | `CUDNN_TF32_MATH_ALLOW_CONVERSION` / TF32 enabled via `CUDNN_NUMERICAL_NOTE_DOWN_CONVERT_INPUTS` exclusion |
| FP16 / BF16 | `CUDNN_TENSOR_OP_MATH` |
| INT8        | `CUDNN_DEFAULT_MATH` |

---

## Call-site inventory

### 1. `src/operator/cudnn_ops.cc` (lines 118-127) — cuDNN v8/v9 frontend path

**Status: CORRECT (active path for sm_120)**

Uses the engine-heuristics numerical-notes API.  TF32 for FP32 is enabled by default:
the code pushes `CUDNN_NUMERICAL_NOTE_DOWN_CONVERT_INPUTS` onto the exclusion list only
when `MXNET_CUDA_TENSOR_OP_MATH_ALLOW_CONVERSION=0`.  This matches PyTorch/TF defaults
on cuDNN 9.  `MXNET_CUDA_ALLOW_TENSOR_CORE` (env, default true on Ampere+) is the outer
gate.  No fix needed.

### 2. `src/operator/nn/cudnn/cudnn_convolution-inl.h` (lines 413-422) — legacy v7 FPA-search path

**Status: STALE but only reachable on CUDNN_VERSION < 7200 builds**

```cpp
cudnnMathType_t math_type = cudnn_tensor_core_ ? CUDNN_TENSOR_OP_MATH : CUDNN_DEFAULT_MATH;
#if CUDNN_VERSION >= 7200
if (GetEnvAllowTensorCore() && GetEnvAllowTensorCoreConversion() &&
    (DataType<DType>::kFlag != kFloat16))
  math_type = CUDNN_TENSOR_OP_MATH_ALLOW_CONVERSION;
#endif
```

- `cudnn_tensor_core_` is only true for FP16 (set at line 73-ish via `DataType<DType>::kFlag == kFloat16`).
- For FP32 on cuDNN 9 the `#if CUDNN_VERSION >= 7200` branch IS compiled, but
  `GetEnvAllowTensorCoreConversion()` returns false by default (see `utils.h:624`
  `MXNET_CUDA_TENSOR_OP_MATH_ALLOW_CONVERSION_DEFAULT false`), so `math_type` stays
  `CUDNN_DEFAULT_MATH` — **no TF32 for FP32 via this path**.
- However on cuDNN 9 this file's `cudnnFindConvolution*` path is a fallback only; the
  primary FP32 path on sm_120 goes through `cudnn_ops.cc` (item 1) which does enable TF32.
  Net effect: harmless for sm_120 unless someone explicitly forces the legacy search path.
- **Lines 613-615** (post-algo-search): re-apply the MathType from `algo.MathType()` which
  returns `CUDNN_TENSOR_OP_MATH` for tensor-core algos and `CUDNN_DEFAULT_MATH` otherwise.
  For FP32 TF32 algos from the v8 path these lines are never reached.

**Verdict**: low-risk stale code; no TF32 regression on sm_120 in practice.

### 3. `src/operator/nn/cudnn/cudnn_deconvolution-inl.h` (lines 427-430) — legacy v7 path

**Status: NEEDS FIX — missing the CUDNN_VERSION >= 7200 / TF32 block**

```cpp
cudnnMathType_t math_type = cudnn_tensor_core_ ? CUDNN_TENSOR_OP_MATH : CUDNN_DEFAULT_MATH;
CUDNN_CALL(cudnnSetConvolutionMathType(forward_conv_desc_, math_type));
CUDNN_CALL(cudnnSetConvolutionMathType(back_conv_desc_,    math_type));
CUDNN_CALL(cudnnSetConvolutionMathType(back_conv_desc_w_,  math_type));
```

Unlike `cudnn_convolution-inl.h`, the deconvolution file has **no** `#if CUDNN_VERSION >= 7200`
TF32/conversion block at all.  For FP32 inputs on cuDNN 9 this hardcodes `CUDNN_DEFAULT_MATH`,
disabling TF32 even if tensor cores are available.

**Proposed fix**: add the same guard that convolution has (see patch in
`.investigations/n23_tf32_patch.patch`).

### 4. `src/operator/nn/cudnn/cudnn_algoreg-inl.h` (line 61) — algo registry helper

```cpp
return IsTensorCoreAlgo() ? CUDNN_TENSOR_OP_MATH : CUDNN_DEFAULT_MATH;
```

This is a read-back helper used when resetting math type after algo selection (convolution
lines 613-615).  Returns correct values for the selected algo; no free-standing call site.
No fix needed.

### 5. `src/operator/rnn-inl.h` (line 1291) — RNN descriptor (cuDNN v8 path)

```cpp
cudnnMathType_t math_type = CUDNN_DEFAULT_MATH;
CUDNN_CALL(cudnnSetRNNDescriptor_v8(..., math_type, ...));
```

The comment above this block reads: "cudnn_tensor_core_ is disabled (see ctor); use default
math for v8."  For RNNs, `CUDNN_DEFAULT_MATH` on cuDNN 9 still allows the backend to pick
TF32 for eligible (FP32) RNN variants — `cudnnSetRNNDescriptor_v8` `mathType` is advisory.
Low priority; leaving as is is safe.
